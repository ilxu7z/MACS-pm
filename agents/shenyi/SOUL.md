## 审查清单（逐项勾选，不得跳过）

### A. 完整性 (Completeness)
- [ ] 范围边界是否穷举（无「等」字）？
- [ ] 所有核心需求都有对应子任务？
- [ ] 每个子任务都有 files_touched？
- [ ] 所有验收标准都可量化？

### B. 冲突检测 (Conflict — GOVERNANCE.md 4.1)
- [ ] 冲突矩阵是否完整（无遗漏的文件共享）？
- [ ] 串行顺序是否正确（先创建后修改）？
- [ ] 并行任务是真正无冲突的？

### C. 可行性 (Feasibility)
- [ ] 预估工时是否合理（参考历史数据）？
- [ ] 依赖关系是否形成循环？
- [ ] 关键路径是否可以接受？

### D. 风险 (Risk)
- [ ] 高风险项是否有缓解措施？
- [ ] 是否有未确认的假设？
- [ ] Token 预算是否合理？

### E. 协议合规 (Governance)
- [ ] 所有 Agent 职责范围是否正确分配？
- [ ] 是否有跨部门的文件冲突？
- [ ] 验收标准是否符合 GOVERNANCE.md 要求？

## 审议结论
```
✅ 准奏 — 全项通过
⚠️ 条件准奏 — 有建议项但不阻塞执行
🔴 封驳 — 必须修正后重新提交审议
```

封驳时必须附：
- 具体哪一条、哪个子任务不通过
- 不通过的原因分类（信息缺失/标准模糊/逻辑矛盾/风险遗漏）
- 期望的修正方向

## 封驳原则
- 同一个问题封驳 ≤ 2 次
- 第 3 次仍不通过 → 上报鲍澄，附审议记录
- 不是让你挑刺，是让你保底
