# DISPATCH.md · 派发前意图评估与询问机制

**适用于：太子（鲍澄）在 sessions_spawn 派发前的决策流程**

## 核心原则

非关键词检测，而是意图理解。收到任务时，太子内部自问三轮判断是否该发起派发询问。

---

## 第一轮：是否属于「适合派发」的任务

| 特征 | 适合派发 | 自己干 |
|------|---------|--------|
| 涉及多文件/多领域 | ✅ i18n 修复（5个JS）、画册（文案+生图+排版） | ❌ 读一个文件、改一个值 |
| 可并行无依赖 | ✅ 翻译+生图+样式同步 | ❌ 串行能搞定的 |
| 需要长时间执行（>10秒） | ✅ 批量生图、翻译、数据抓取 | ❌ ls、grep、git status |
| 不确定最优方案 | ✅ 探索性分析、多技术路线比较 | ❌ 单一方案明确 |
| 会产出新文件 | ✅ 图片、文档、HTML | ❌ 改现有文件的某一处 |

**规则：** 命中任意一行「适合派发」→ 进入第二轮。否则自己干，不问。

## 第二轮：构建推荐方案

确定要派发后，向老板输出询问消息，格式如下：

```
📋 检测到此任务适合子 Agent 执行，推荐方案：

  ▸ 拆分为 [N] 个子任务并行执行
  ▸ 预计节省时间：~X%
  ▸ 分拆概要：[子任务1] / [子任务2] / ...

  [Y/n] 启用派发模式？
```

**格式规则：**
- 简短（不超过 5 行）
- 给出具体拆分建议，不空问「要不要派」
- 不接受则自己干，不再提
- 接受则批量 spawn，完成后统一审查

## 第三轮：尊重否定与回弹

| 老板回应 | 行动 |
|---------|------|
| 「不派」 | 不啰嗦，直接自己干 |
| 「分法不对」 | 按指示调整后再派 |
| 「全派出去」 | 不留尾巴，全部 spawn |
| 「自己来」 | 尊重决定，全程自己干 |

## 与现有机制的衔接

- **三省六部六道防线**（GLOBAL.md/Layer 1→6）：意图评估发生在防线之前——太子收到任务后先判断是否自己干/派发/问老板，确定派发后再走筹微→审微→驿使→执行的防线流程。
- **GOVERNANCE.md 协作纪律**：DISPATCH.md 管「要不要派」，GOVERNANCE.md 管「派了以后怎么管」。
- **GLOBAL.md 通信规则**：派发后子 Agent 遵循「只在被 sessions_spawn 调用时工作」的规则。

## 不触发询问的场景

- **纯问答咨询**（「现在几点」「这条信息什么意思」）→ 不问
- **老板明确指令**（「帮我改」「直接做」「快改」）→ 不问，直接执行
- **Design Read 自动激活** — 技能调用规则与意图评估不冲突，意图评估跑在技能激活之前
- **三省六部自动巡检**（cron 任务）→ 已预先配置好调度策略，无需询问
