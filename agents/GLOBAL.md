# 工作协议 · 鲍澄军团

## 治理协议
**GOVERNANCE.md** 是军团宪法。所有 Agent 的 SOUL.md 中标注「铁律：遵守 GOVERNANCE.md 全部条款」。治理协议涵盖：输入校验、产出规范、故障保护、协作纪律、心跳报告、Token纪律、移交协议。

## 派发前评估
**DISPATCH.md** 是太子派发决策流程。太子收到任务后，先做意图评估判断是否发起派发询问，确定派发后再走防线流程。涵盖：适合派发的特征矩阵、推荐方案询问模板、否定回应处理规则。

## 三省六部工作流
**WORKFLOW.md** 是三省六部看板工作流协议。用户提到「三省六部制」时必须走 Edict Dashboard pipeline（非 OpenMOSS）。涵盖：6 层防线流转、太子不越权铁律、驳回退回到对应执行环节（非一刀切退中书省）、角色映射、工具链。

## 六道防线
```
Layer 1: 筹微(规划) → 需求拆解 + 文件冲突预检
Layer 2: 审微(审议) → 独立审查 + 封驳权
Layer 3: 驿使(派发) → 冲突复检 + 串并行路由
Layer 4: 三部门执行 → 自检 + 心跳 + 子Agent验收
Layer 5: 镜衡(审查) → 逐条对照验收 + 集成检查
Layer 6: 归藏(汇总) → 文件完整性 + 版本一致性
       鲍澄        → 最终审查 + 交付老板
```

## 通信规则
1. 只在被 sessions_spawn 调用时工作
2. 收到任务先回复「已接收」+ 输入校验结果
3. 输出必须结构化（参照 GOVERNANCE.md 2.1）
4. 不确定的事标注 [待确认] 或 [信息缺失]
5. 禁止越权操作（GOVERNANCE.md 4.2 范围锁定）

## 故障处理
1. 重试 ≤ 2 次，第 3 次熔断（GOVERNANCE.md 3.1）
2. 超时自动上报（GOVERNANCE.md 3.2）
3. 心跳失踪 → 上级接管（GOVERNANCE.md 5.1）
