# GOVERNANCE.md · 鲍澄军团治理协议

所有 Agent 必须遵守。违反则产出无效，打回重做。

---

## 一、输入校验

### 1.1 任务接收契约
每个 Agent 接收 Work Package 时必须验证：
- [ ] 任务ID 存在且唯一
- [ ] 任务描述包含明确的「做什么」和「不做什么」
- [ ] 验收标准可量化（禁止"做好""完善"等模糊词）
- [ ] 上下文信息完整（依赖的产出物路径/版本标注）

### 1.2 缺信息处理
若输入不完整：
- ❌ 禁止猜测、禁止假定缺省值
- ✅ 必须在回复中标注 `[信息缺失]` 并列出具体缺什么
- ✅ 返回给上级（鲍澄或部门 Agent），等待补全后重新下发

---

## 二、产出规范

### 2.1 强制结构
所有 Agent 产出必须为结构化 Markdown，包含：
```
## 任务ID: [ID]
## 产出摘要
## 产出详情
## 文件清单（如有）
## 自检结果
## 已知限制与风险
```

### 2.2 files_touched（必填）
任何会修改/创建文件的 Agent 必须在产出头部声明：
```
files_touched:
  - path/to/file.html  (读/写/创建)
  - path/to/file.css   (修改)
```
未声明而修改了文件 = 视为越权，验收不通过。

### 2.3 版本标注
引用其他 Agent 的产出时，必须注明版本/时间戳：
```
依赖:
  - 设计规范 v1 (绘象 @ 2026-05-28 10:00)
  - 首页文案 v2 (墨卿 @ 2026-05-28 11:00)
```
引用过期版本 = 验收不通过。

---

## 三、故障保护

### 3.1 重试与熔断
- 同一策略失败 ≤ 2 次可重试
- 第 3 次失败 → 停止重试，标注 `[熔断]` 并报告根因分析
- 禁止无变更重试（改超时时间不算变更）

### 3.2 超时处理
- 单次执行 > 预估工时 × 2 → 标记 `[超时]`，向上报告
- 子 Agent 超时 → 部门 Agent 接管，不可无限等待

### 3.3 幻觉检测
以下模式触发警报，自动标注为可疑：
- 引用不存在文件路径
- 声称的数据无法从输入文档中找到
- 输出中出现「据分析」「据统计」等但未标注数据源

---

## 四、协作纪律

### 4.1 文件冲突矩阵
在规划/派发阶段检测，规则：
- 两个并行 Agent 的 files_touched 有交集 → 必须串行
- 串行顺序：先「创建」的在前，后「修改」的在后
- 只有一个 Agent 声明该文件 → 无冲突，可并行

### 4.2 范围锁定
- 只做 Work Package 里声明的事
- 额外发现的问题 → 记录在「已知限制」中，等待下轮规划纳入
- 禁止「顺手改一下」

### 4.3 审批链不可跳跃
每个审批节点必须获得明确「通过」后，才进入下一节点：
- 筹微产出 → 审微审议 → 鲍澄拍板 → 驿使派发
- 跳过任一环节 = 流程违规，产出无效

---

## 五、状态完整性

### 5.1 心跳报告
每个子 Agent 执行中每 5 分钟（或完成一个子模块后）向上级 Agent 汇报：
```
[心跳] 任务ID: [ID] | 进度: [X/Y] | 当前: [正在做什么] | 预估剩余: [时间]
```
连续 2 个心跳窗口未收到 → 上级 Agent 标记 `[失联]`，启动接管流程。

### 5.2 卡死检测
驿使的派发指令中包含 deadline。超时未完成：
- 第 1 次超时：自动提醒
- 第 2 次超时：上报鲍澄，决定是否换 Agent 或调整方案

## 六、Token 纪律

- 规划/审议/审查类：优先 DeepSeek V4 Pro（高性价比推理）
- 代码开发：Claude Sonnet 优先，Opus 仅在复杂场景
- 生图：Gemini 3 Pro，单个任务 ≤ 5 次调用
- 禁止为同一结果反复调用昂贵模型

---

## 七、移交协议

Agent A 将任务转交给 Agent B 时：
1. A 产出一个 `handoff.json`：
   ```json
   {
     "task_id": "...",
     "completed": ["已完成的子项"],
     "pending": ["未完成的原因"],
     "blockers": ["阻塞项"],
     "context": "简要上下文"
   }
   ```
2. B 必须读完 handoff 再开始执行
3. 禁止直接转发 A 的输出而不读

---

## 八、防线分层

```
Layer 1: 筹微(规划)     → 拆解时检测文件冲突，标注边界
Layer 2: 审微(审议)     → 审查 Layer1 是否遗漏、验收标准是否可量化
Layer 3: 驿使(派发)     → 再次检测冲突矩阵，确定串并行
Layer 4: 部门Agent       → 执行中自检 + 心跳 + 子Agent验收
Layer 5: 镜衡(审查)      → 对照标准逐项验收 + 集成检查 + 文件冲突检查
Layer 6: 归藏(汇总)      → 文件完整性 + 版本一致性 + 部署可行性
Layer 7: 鲍澄(最终审查)  → 全局过一眼，交付老板
```

**每一层都有否决权。任何一层发现的问题必须在当前层解决，不传递到下一层。**
